# 지선차단기 복구 플랜 정리

작성일: 2026-04-06

## 1. 목적

지선차단기의 차단 이후 복구 방식을 현재까지 논의된 기준으로 정리한다.

## 2. 기본 원칙

- 수동 복구는 적용하지 않는다.
- 원격 복구는 반드시 지원한다.
- 보호회로 차단 후 복구는 서버 명령으로만 수행한다.
- 대기전력 차단 후 복구는 `스케줄 복구`와 `원격 복구`를 적용한다.
- 지선차단기는 동작 후 상태를 즉시 push 전송하지 않는다.
- DCU가 polling으로 결과를 확인한다.

## 3. 복구 방식 구분

### 3.1 보호회로 차단 후 복구

- 과전류, 과전압 등 보호 조건으로 차단된 경우
- 서버가 별도 복구 명령을 내려야 한다
- 자동 복구는 하지 않는다

### 3.2 대기전력 차단 후 복구

- 절전 목적의 차단 상태
- 아래 두 가지 방식으로 복구한다
  - 스케줄 복구
  - 원격 복구

## 4. 스케줄 복구

### 4.1 개념

- 패턴이 명확한 곳에서 사용
- 항상 특정 시간에 다시 켜지도록 구성
- DCU가 특정 시간마다 하위 지선차단기에 `relay_on` 명령을 보낸다

### 4.2 플로우

1. 지선차단기가 대기전력 기준에 의해 차단된다
2. DCU는 해당 지선차단기의 복구 스케줄을 보유한다
3. 설정된 시간 도달 시 DCU가 `relay_on` 명령을 보낸다
4. 지선차단기는 릴레이 ON 동작을 수행한다
5. 지선차단기는 즉시 상태를 전송하지 않는다
6. 다음 polling 시 DCU가 계측값과 명령 반영 여부를 확인한다
7. 기대값과 다르면 재요청 또는 오류 처리한다

## 5. 원격 복구

### 5.1 개념

- 관리자 또는 서버가 복구 기능을 수행
- 서버 또는 관리자 요청에 따라 DCU가 하위 지선차단기에 `relay_on` 명령을 보낸다

### 5.2 플로우

1. 관리자 또는 서버가 복구 요청을 한다
2. 서버가 DCU에 복구 명령을 전달한다
3. DCU가 대상 지선차단기에 `relay_on` 명령을 보낸다
4. 지선차단기는 릴레이 ON 동작을 수행한다
5. 지선차단기는 즉시 상태를 전송하지 않는다
6. 다음 polling 시 DCU가 계측값과 명령 반영 여부를 확인한다
7. 기대값과 다르면 재요청 또는 오류 처리한다

## 6. 결과 확인 방식

- 지선차단기는 명령 수행 후 상태를 바로 전송하지 않는다
- DCU가 주기적으로 polling 한다
- polling 결과가 기대값과 일치하는지 확인한다
- 기대값과 다르면 재요청하거나 오류로 처리한다

## 7. 설계상 핵심

- 지선차단기의 별도 상태값은 두지 않는다
- `relay_on`, `relay_off` 명령 자체가 상태 의미를 가진다
- 명령 성공 여부는 push 응답이 아니라 polling 결과로 판단한다

## 8. 정리

- 수동 복구: 미적용
- 원격 복구: 필수
- 보호회로 차단 후 복구: 서버 명령 기반
- 대기전력 차단 후 복구: 스케줄 복구 + 원격 복구
- 결과 검증: DCU polling 기반

## 9. 통신 관련 설명

- 대기전력 차단이 발생해도 UART 통신은 계속 가능하다.
- 연결 구조는 `DCU - 지선차단기 MCU - 계량칩 - 릴레이` 순서로 본다.
- 릴레이 차단은 릴레이 후단 부하측 전원을 끊는 방식이므로 MCU 전원은 유지된다.
- 따라서 대기전력 차단 상태에서도 지선차단기 MCU는 살아있고 DCU와의 통신을 유지할 수 있다.
- 계량칩과 MCU가 살아있는 구조이므로 polling, `relay_on`, `relay_off` 명령 처리는 계속 가능하다.

## 10. TBD 검토 사항

- 과전류 또는 과전압 발생 시 지선차단기 MCU가 정상 동작을 유지할 수 있는지는 HW 보호회로 설계와 전원부 구성에 따라 달라진다.
- 따라서 MCU 전원 입력부, 계량칩 입력부, UART/RS-485 통신부에 대한 서지 및 이상전압 보호 설계 검토가 필요하다.
- 해당 내용은 회로도 수령 후 HW팀과 별도 확인이 필요하다.
