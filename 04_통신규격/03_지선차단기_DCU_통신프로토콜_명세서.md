# 지선차단기 - DCU 통신 프로토콜 명세서

작성일: 2026-04-10

## 1. 목적

본 문서는 지선차단기와 DCU 간의 초기 버전 통신 프로토콜을 정의한다.

본 프로토콜의 목표는 다음과 같다.

- 지선차단기의 계측값과 상태 정보를 DCU가 안정적으로 수집할 수 있어야 한다.
- DCU가 릴레이 ON/OFF 제어를 안전하게 요청할 수 있어야 한다.
- 통신 구조는 단순하고 디버깅이 쉬워야 한다.
- 향후 데이터 항목 추가가 가능해야 한다.

본 문서는 초기 버전의 기준 문서이며, 펌웨어, DCU, 시험 항목의 공통 기준으로 사용한다.

## 2. 적용 범위

본 명세서는 아래 통신 구간에 적용한다.

- DCU <-> 지선차단기

이번 버전에서 포함하는 데이터는 아래와 같다.

- `SL_ID`
- `Voltage`
- `Current`
- `Energy_Acc`
- `Relay_State`
- `Alarm_Code`

이번 버전에서 제외하는 항목은 아래와 같다.

- 지선차단기 장치 시간
- 시간 동기화
- 완전한 DLMS/COSEM 세션 구조
- 인증서 기반 보안

지선차단기 장치 시간을 제외하는 이유는 아래와 같다.

- 수집 기준 시간은 DCU 수신 시각을 사용한다.

## 3. 통신 대상과 역할

- `DCU`: Client, Master
- `지선차단기`: Server, Slave

동작 원칙은 아래와 같다.

- DCU가 요청을 보낸다.
- 지선차단기가 응답을 보낸다.
- 릴레이 제어는 DCU가 명령하고, 지선차단기가 수행 결과와 실제 상태를 응답한다.

## 4. 통신 방식 개요

### 4.1 기본 방식

초기 버전은 간단한 전용 요청/응답 프로토콜을 사용한다.

- 조회는 `Read Request` / `Read Response`로 처리한다.
- 제어는 `Control Request` / `Control Response`로 처리한다.
- 필요 시 향후 `Event Report`를 추가할 수 있도록 메시지 코드를 확보한다.

### 4.2 설계 이유

초기 버전에서 간단한 전용 프로토콜을 채택한 이유는 아래와 같다.

- 현재 필요한 데이터 종류가 많지 않다.
- 운영 핵심이 표준 호환보다 수집과 제어 확인에 있다.
- MCU 구현 부담을 줄일 수 있다.
- DCU 파싱 로직이 단순해진다.
- 현장 시험 및 디버깅이 쉽다.

## 5. 프레임 구조

### 5.1 기본 프레임 형식

| 필드 | 크기 | 설명 |
|---|---:|---|
| STX | 1 byte | 프레임 시작 표시 |
| VER | 1 byte | 프로토콜 버전 |
| MSG_TYPE | 1 byte | 메시지 종류 |
| SEQ | 1 byte | 요청-응답 식별 번호 |
| SL_ID | 1 byte | 장치 식별자 |
| LEN | 2 byte | Payload 길이 |
| PAYLOAD | N byte | 실제 데이터 |
| CRC16 | 2 byte | 프레임 오류 검출 |
| ETX | 1 byte | 프레임 종료 표시 |

### 5.2 필드 정의

#### STX

- 프레임 시작 위치를 구분하기 위한 값이다.
- 수신 중 깨진 프레임 이후 재동기를 쉽게 하기 위해 필요하다.
- 초기 버전 고정값은 `0x7E`로 정의한다.

#### VER

- 프로토콜 버전 식별용이다.
- 향후 항목 추가 또는 구조 변경 시 호환성을 관리하기 위해 필요하다.

#### MSG_TYPE

- 조회 요청, 조회 응답, 제어 요청, 제어 응답을 구분하기 위해 사용한다.

#### SEQ

- 요청과 응답을 짝지어 확인하기 위해 사용한다.
- timeout 이후 재전송이 발생해도 중복 응답을 구분할 수 있다.
- 릴레이 제어처럼 중요한 명령에서 이중 수행 위험을 줄인다.

#### SL_ID

- 어떤 지선차단기에 대한 요청 또는 응답인지 식별하기 위해 사용한다.
- 현장 설치, 교체, 유지보수 이력의 기준이 된다.

#### LEN

- PAYLOAD 길이를 나타낸다.
- 가변 길이 데이터 구조에서 수신 안정성을 높인다.

#### PAYLOAD

- 실제 계측값, 상태값, 명령값, 응답코드를 담는 영역이다.

#### CRC16

- 통신 중 발생하는 비트 오류와 프레임 손상을 검출하기 위해 사용한다.
- 잘못된 값 해석이나 잘못된 제어 수행 위험을 줄인다.

#### ETX

- 프레임 종료 위치를 구분하기 위한 값이다.
- 시작 바이트와 함께 프레임 경계를 명확히 하기 위해 필요하다.
- 초기 버전 고정값은 `0x7E`로 정의한다.

## 6. 메시지 종류

| 코드 | 이름 | 설명 |
|---|---|---|
| `0x01` | Read Request | 데이터 조회 요청 |
| `0x02` | Read Response | 데이터 조회 응답 |
| `0x03` | Control Request | 릴레이 제어 요청 |
| `0x04` | Control Response | 릴레이 제어 응답 |
| `0x05` | Event Report | 향후 확장용 이벤트 보고 |
| `0x06` | ACK | 정상 수신 확인 |
| `0x07` | NACK | 오류 또는 거부 응답 |

## 7. Payload 구조

### 7.1 기본 형식

Payload는 아래의 TLV 유사 구조를 사용한다.

| 필드 | 크기 | 설명 |
|---|---:|---|
| ITEM_ID | 2 byte | 데이터 항목 ID |
| TYPE | 1 byte | 데이터 타입 |
| ITEM_LEN | 1 byte | VALUE 길이 |
| VALUE | N byte | 실제 값 |

### 7.2 사용 이유

- 항목 추가가 쉬워진다.
- 수신 측이 값의 타입과 길이를 명확히 알 수 있다.
- 여러 항목을 한 프레임 안에 담을 수 있다.

### 7.3 3개 항목 단위 전송 규칙

초기 버전에서는 한 번의 응답 Payload에 최대 3개의 정보 항목을 담는다.

- 1프레임 최대 데이터 항목 수: `3개`
- 요청 항목이 4개 이상이면 `3개 + 나머지` 형태로 나누어 송신한다.
- 마지막 묶음이 3개보다 적으면 남는 자리는 `data_empty(0x0000)`로 채운다.

예:

- 4개 항목 응답 시: `3개 / 1개 + 0x0000 + 0x0000`
- 5개 항목 응답 시: `3개 / 2개 + 0x0000`
- 6개 항목 응답 시: `3개 / 3개`

## 8. 데이터 항목 정의

### 8.1 지선차단기 -> DCU 송신 항목

| 항목 ID | 항목명 | 타입 | 단위 | 필수 여부 | 설명 |
|---|---|---|---|---|---|
| `0x0000` | data_empty | uint32 | - | 패딩 | 빈 슬롯 채움용 |
| `0x0001` | SL_ID | uint8 | - | 필수 | 장치 식별자 |
| `0x0002` | Voltage | uint32 | 0.1 V | 필수 | 실효 전압 |
| `0x0003` | Current | uint32 | 0.01 A | 필수 | 실효 전류 |
| `0x0004` | Energy_Acc | uint32 | Wh | 필수 | 누적 전력량 |
| `0x0005` | Relay_State | uint8 | - | 필수 | 릴레이 현재 상태 |
| `0x0006` | Alarm_Code | uint16 bit field | - | 권장 | 이상 상태 코드 |

### 8.2 DCU -> 지선차단기 송신 항목

| 항목 ID | 항목명 | 타입 | 필수 여부 | 설명 |
|---|---|---|---|---|
| `0x1001` | Read_Item_List | list | 필수 | 조회할 항목 목록 |
| `0x1002` | Relay_Command | uint8 | 필수 | 릴레이 ON/OFF 명령 |
| `0x1003` | Control_Reason | uint8 | 권장 | 제어 사유 코드 |
| `0x1004` | Control_Token | uint32 | 권장 | 제어 명령 검증값 |

## 9. 데이터 항목별 채택 이유

### 9.1 data_empty

- 3개 단위 전송 규칙에서 남는 슬롯을 채우기 위한 예약 항목이다.
- 마지막 프레임의 형식을 고정적으로 유지하는 데 도움이 된다.

### 9.2 SL_ID

- 데이터의 장치 출처를 식별하는 기본 키다.
- 서버, DCU, 현장 유지보수 인력이 같은 장치를 동일하게 인식할 수 있게 한다.

### 9.3 Voltage

- 저전압, 과전압, 무전압 등 전원 상태를 확인할 수 있다.
- 계통 이상과 장치 이상을 구분하는 기초 정보가 된다.

### 9.4 Current

- 실제 부하 동작 상태를 파악할 수 있다.
- 과전류 또는 무부하 상태 판단에 필요하다.
- 릴레이 제어 후 실제 차단 또는 투입 여부를 간접적으로 검증할 수 있다.

### 9.5 Energy_Acc

- 누적 사용량 확인과 장기 이력 분석에 필요하다.
- 일부 통신 누락이 있어도 누적값 기준으로 추적이 가능하다.

### 9.6 Relay_State

- 제어 명령 결과가 실제로 반영되었는지 확인하는 핵심 항목이다.
- "OFF 명령을 보냈다"와 "실제로 OFF 상태가 되었다"는 다르기 때문에 반드시 필요하다.

### 9.7 Alarm_Code

- 전압, 전류 수치만으로 설명되지 않는 내부 이상 상태를 전달하기 위한 항목이다.
- 릴레이 구동 실패, 측정 오류, 메모리 이상 등을 빠르게 구분할 수 있다.

### 9.8 Read_Item_List

- DCU가 필요한 항목만 선택 조회할 수 있게 한다.
- 통신량을 줄이고 상황별 조회 구성을 유연하게 할 수 있다.

### 9.9 Relay_Command

- 원격 ON/OFF 제어의 핵심 명령 항목이다.

### 9.10 Control_Reason

- 수동 제어, 스케줄 제어, 보호 제어 등 제어 사유를 구분하는 데 유용하다.
- 운영 로그 해석에 도움이 된다.

### 9.11 Control_Token

- 복잡한 보안을 모두 구현하지 않더라도 최소한의 명령 검증 수단으로 사용할 수 있다.
- 오동작 또는 비정상 명령 유입 방지에 도움이 된다.

## 10. 데이터 타입 및 스케일

| TYPE 예시 | 의미 |
|---|---|
| `0x01` | uint8 |
| `0x02` | uint16 |
| `0x03` | uint32 |
| `0x04` | int32 |
| `0x05` | bit field |
| `0x06` | list |

스케일 정의는 아래와 같다.

- `Voltage = raw / 10`
- `Current = raw / 100`
- `Energy_Acc = raw 그대로 Wh`

예시:

- `Voltage = 2234` 이면 `223.4 V`
- `Current = 1050` 이면 `10.50 A`

### 10.1 바이트 수 및 스케일 권장 근거

| 구분 | 평균 데이터량 예시 | 권장 바이트 | 보정치 / 파싱 방법 |
|---|---|---:|---|
| 전압 | 약 `220.0 ~ 230.0V` | `4 byte` | `uint32`, 실제값 = `raw / 10` |
| 전류 | 약 `0.00 ~ 50.00A` | `4 byte` | `uint32`, 실제값 = `raw / 100` |
| 전력량 | 세대 월 사용량 약 `150 ~ 310kWh`, 큰 사용량 `900 ~ 1,700kWh`, 공용부 예시 `25,000kWh/월` | `4 byte` | `uint32`, 실제값 = `raw Wh` |

전력량은 누적값이므로 가장 큰 범위를 고려해야 한다.

220V, 50A 정격 기준 보수 계산은 아래와 같다.

- 최대 전력: `220 x 50 = 11kW`
- 월 최대: 약 `7,920kWh`
- 연 최대: 약 `96,360kWh`
- 10년 누적: 약 `963,600kWh`
- 20년 누적: 약 `1,927,200kWh`

따라서 초기 버전 계측값은 전압, 전류, 전력량 모두 `4 byte`로 통일한다.

## 11. 상태값 및 알람코드 정의

### 11.1 릴레이 상태 정의

| 값 | 의미 |
|---|---|
| `0x00` | OFF / 차단 |
| `0x01` | ON / 투입 |
| `0x02` | 전환 중 |
| `0x03` | 이상 또는 판별 불가 |

### 11.2 릴레이 상태를 별도로 두는 이유

- 제어 성공 여부를 실제 상태로 확인하기 위해 필요하다.
- 접점 이상, 용착, 구동 실패 등은 명령 응답만으로는 확인이 어렵다.

### 11.3 알람 코드 정의

초기 버전에서는 16비트 bit field 방식을 사용한다.

| 비트 | 의미 |
|---:|---|
| bit0 | 과전류 |
| bit1 | 과전압 |
| bit2 | 저전압 |
| bit3 | 릴레이 구동 실패 |
| bit4 | 측정칩 이상 |
| bit5 | 내부 메모리 이상 |
| bit6 | 센서 또는 측정값 이상 |
| bit7 | 예약 |
| bit8~15 | 향후 확장 |

### 11.4 알람 코드를 두는 이유

- 수치값만으로는 이상 원인을 빠르게 판단하기 어렵다.
- DCU가 경보 정책을 바로 적용할 수 있다.
- 유지보수 시 원인 파악 속도를 높일 수 있다.

### 11.5 과전압 / 과전류 정의

- 과전압: `220V 기준 ±10%` 범위를 벗어나는 경우
- 과전압 판정 범위:
  - 저한계 미만: `198V 미만`
  - 고한계 초과: `242V 초과`
- 과전류: `N회 측정값의 Max값보다 M% 높은 경우`

주의:

- `N`, `M` 값은 운용 및 시험 기준에 따라 별도 확정한다.
- 본 문서에서는 개념 정의만 우선 반영한다.

## 12. 응답 코드 정의

| 코드 | 의미 |
|---|---|
| `0x00` | 성공 |
| `0x01` | 알 수 없는 메시지 |
| `0x02` | 지원하지 않는 항목 |
| `0x03` | 길이 오류 |
| `0x04` | CRC 오류 |
| `0x05` | 잘못된 값 |
| `0x06` | 제어 거부 |
| `0x07` | 내부 처리 실패 |
| `0x08` | 토큰 또는 인증 오류 |

응답 코드는 실패 원인을 빠르게 구분하기 위해 필요하다.

## 13. 통신 시나리오

### 13.1 주기 조회

DCU는 주기적으로 아래 항목을 조회한다.

- `SL_ID`
- `Voltage`
- `Current`
- `Energy_Acc`
- `Relay_State`
- `Alarm_Code`

### 13.2 조회 요청 예시

전압, 전류, 릴레이 상태 조회 예시:

```text
[7E][01][01][21][SL_ID][0006][0002 0003 0005][CRC][7E]
```

### 13.3 조회 응답 예시

전압 223.4V, 전류 10.50A, 릴레이 ON 응답 예시:

```text
[7E][01][02][21][SL_ID][0011]
[0002][03][04][00 00 08 BA]
[0003][03][04][00 00 04 1A]
[0005][01][01][01]
[CRC][7E]
```

### 13.4 제어 요청 예시

릴레이 OFF 제어 요청:

```text
[7E][01][03][22][SL_ID][0002][1002 00][CRC][7E]
```

### 13.5 제어 응답 예시

릴레이 OFF 제어 성공 및 실제 상태 OFF 응답:

```text
[7E][01][04][22][SL_ID][0004][00][0005 01 00][CRC][7E]
```

의미는 아래와 같다.

- `00`: 성공
- `Relay_State = 00`: 실제 릴레이 OFF

## 14. timeout / retry / 중복 처리 정책

초기 권장 정책은 아래와 같다.

- DCU가 SL로 요청 전송
- 응답 대기 방식 또는 비동기식 전송 방식 사용
- 응답 timeout: `1초`
- 응답 미수신 시 `최대 3회 재전송`
- 총 3회 response 수신 실패 시 서버에 event 발생 알림

중복 처리 정책은 아래와 같다.

- 동일 장치에 동일 `SEQ`의 제어 요청이 재수신되면 중복 요청으로 판단한다.
- 이미 처리된 요청이면 이전 결과를 재응답하거나 중복 수행을 방지한다.

### 14.1 SEQ 정책

- `SEQ`는 1 byte 순환값을 사용한다.
- 범위는 `0x00 ~ 0xFF`이다.
- 송신 시 1씩 증가한다.
- `0xFF` 다음은 `0x00`으로 순환한다.

## 15. 버전 정책 및 확장 규칙

- 초기 버전은 `VER = 0x01`로 정의한다.
- 신규 항목 추가는 기존 항목 ID를 유지한 상태에서 뒤에 추가한다.
- 기존 의미 변경이 필요한 경우 버전 증가를 검토한다.
- 향후 `Instant_Power`, `FW_Version`, `Device_Status`, `Event Report`를 추가할 수 있다.
