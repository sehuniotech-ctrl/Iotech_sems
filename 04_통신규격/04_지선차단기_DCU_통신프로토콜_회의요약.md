# 지선차단기 - DCU 통신 프로토콜 회의 요약

작성일: 2026-04-10

## 1. 이번 회의에서 정할 내용

- 초기 버전 통신 프로토콜은 간단한 전용 요청/응답 방식으로 간다.
- 장치 시간은 이번 버전에서 제외한다.
- DCU 수신 시각을 기준 시간으로 사용한다.
- 필수 데이터는 `SL_ID`, `Voltage`, `Current`, `Energy_Acc`, `Relay_State`로 정한다.
- `Alarm_Code`는 권장 포함 항목으로 넣는다.
- 프레임 시작/종료 바이트는 `0x7E`를 사용한다.
- `SL_ID`는 `1 byte`로 사용한다.
- 전압, 전류, 전력량은 모두 `4 byte`로 통일한다.
- 한 번에 최대 3개 항목씩 전송하고, 남는 슬롯은 `data_empty(0x0000)`로 채운다.

## 2. 왜 간단한 방식으로 가는가

- 현재 필요한 데이터 종류가 많지 않다.
- 핵심 목적은 표준 호환보다 안정적인 수집과 릴레이 제어 확인이다.
- MCU와 DCU 구현 부담을 줄일 수 있다.
- 현장 디버깅과 시험 대응이 쉽다.
- 향후 항목 추가도 가능하다.

## 3. 왜 DLMS/COSEM을 그대로 쓰지 않는가

- DLMS/COSEM은 표준성은 좋지만 구조가 무겁다.
- 링크 연결, 세션 수립, 객체 접근 구조까지 구현해야 한다.
- 현재 요구 범위에 비해 개발과 시험 부담이 크다.
- 지금은 표준 완전 호환보다 빠르고 명확한 구현이 더 중요하다.

## 4. 프로토콜 구조 핵심

프레임은 아래 구조를 사용한다.

```text
[7E][VER][MSG_TYPE][SEQ][SL_ID][LEN][PAYLOAD][CRC16][7E]
```

이 구조를 쓰는 이유는 아래와 같다.

- `MSG_TYPE`: 조회인지 제어인지 구분하기 위해 필요하다.
- `SEQ`: 요청과 응답을 맞추고 재전송 중복을 막기 위해 필요하다.
- `SL_ID`: 어느 장치의 데이터인지 식별하기 위해 필요하다.
- `LEN`: 데이터 길이를 정확히 알기 위해 필요하다.
- `CRC16`: 통신 오류를 검출하기 위해 필요하다.

## 5. 꼭 넣어야 하는 데이터와 이유

### 5.1 SL_ID

- 어떤 장치의 데이터인지 식별하기 위해 필요하다.

### 5.2 Voltage

- 전원 상태와 이상 여부를 판단하기 위해 필요하다.

### 5.3 Current

- 실제 부하 동작 상태와 과전류 여부를 보기 위해 필요하다.

### 5.4 Energy_Acc

- 누적 사용량 확인과 이력 추적에 필요하다.

### 5.5 Relay_State

- 제어 명령이 실제로 반영되었는지 확인하기 위해 필요하다.
- "명령을 보냈다"와 "실제 상태가 바뀌었다"는 다르기 때문이다.

### 5.6 Alarm_Code

- 전압, 전류 값만으로 설명되지 않는 내부 이상 원인을 전달하기 위해 필요하다.
- 릴레이 구동 실패, 측정칩 이상, 메모리 이상 등을 구분할 수 있다.

## 6. DCU가 보내는 데이터

- `Read_Item_List`: 어떤 항목을 읽을지 지정
- `Relay_Command`: 릴레이 ON/OFF 제어
- `Control_Reason`: 왜 제어했는지 구분
- `Control_Token`: 최소한의 제어 검증값

## 6. 추가 결정사항

- 과전압 정의: `220V 기준 ±10%` 범위를 벗어난 경우
- 과전류 정의: `N회 측정값의 Max값보다 M% 높은 경우`
- DCU는 요청 후 응답 대기 또는 비동기 수신 방식 사용 가능
- 응답이 없으면 `최대 3회 재전송`
- 3회 응답 실패 시 서버에 event 발생 알림

## 7. 회의에서 말할 핵심 문장

- 초기 버전은 표준 전체 구현보다 필요한 기능을 빠르고 명확하게 구현하는 것이 중요하다.
- 그래서 DLMS의 확장 개념은 참고하되, 실제 프로토콜은 경량 전용 방식으로 간다.
- 제어 신뢰성을 위해 `Relay_State`, 장애 해석을 위해 `Alarm_Code`, 요청-응답 매칭을 위해 `SEQ`를 포함한다.
- 지금은 단순하고 확실하게 만들고, 이후 필요 시 확장하는 방향이 맞다.
