# 지선차단기 - DCU 통신 프로토콜 2안 비교

작성일: 2026-04-10

## 1. 목적

본 문서는 지선차단기와 DCU 간 통신 프로토콜을 두 가지 방식으로 제안하고, 각 방식의 특징과 채택 이유를 비교하기 위한 회의용 자료이다.

비교 대상은 아래와 같다.

- 1안: 간단한 전용 프로토콜 방식
- 2안: DLMS/COSEM 기반 방식

두 안 모두 아래 데이터를 대상으로 한다.

- `SL_ID`
- `Voltage`
- `Current`
- `Energy_Acc`
- `Relay_State`
- `Alarm_Code`

이번 비교안에서는 지선차단기 장치 시간은 제외한다.

- 이유: 시간 기준은 DCU 수신 시각을 사용한다.

## 2. 결론 요약

현재 요구 범위와 개발 효율을 기준으로 보면, 초기 버전은 `1안: 간단한 전용 프로토콜 방식`이 더 적합하다.

그 이유는 다음과 같다.

- 필요한 데이터 종류가 많지 않다.
- 원격 제어와 상태 확인이 핵심이다.
- MCU 및 DCU 개발 복잡도를 낮출 수 있다.
- 시험, 디버깅, 유지보수가 쉽다.

반면, 향후 외부 계량 표준 연동이나 표준 인증, 타 시스템 호환이 매우 중요해진다면 `2안: DLMS/COSEM 기반 방식`이 장기적으로 유리할 수 있다.

## 3. 공통 요구사항

두 안 모두 아래 요구를 만족해야 한다.

- DCU가 지선차단기 데이터를 주기적으로 조회할 수 있어야 한다.
- DCU가 릴레이 ON/OFF 제어를 수행할 수 있어야 한다.
- 지선차단기가 현재 릴레이 상태를 반드시 응답해야 한다.
- 통신 오류를 검출할 수 있어야 한다.
- 향후 데이터 항목 추가가 가능해야 한다.

## 4. 1안: 간단한 전용 프로토콜 방식

### 4.1 개념

간단한 전용 프로토콜 방식은 지선차단기와 DCU가 필요한 데이터만 주고받도록 최소한의 규칙으로 정의한 경량 프로토콜이다.

전체 구조는 아래 원칙을 따른다.

- DCU가 요청한다.
- 지선차단기가 응답한다.
- 데이터는 항목 ID 기반으로 주고받는다.
- 제어 결과는 상태값으로 다시 확인한다.

즉, DLMS의 복잡한 세션과 객체 모델은 생략하고, 실사용에 필요한 통신 기능만 남기는 방식이다.

### 4.2 기본 프레임

| 필드 | 크기 | 설명 |
|---|---:|---|
| STX | 1 byte | 프레임 시작 |
| VER | 1 byte | 프로토콜 버전 |
| MSG_TYPE | 1 byte | 요청/응답/제어 구분 |
| SEQ | 1 byte | 요청-응답 식별 번호 |
| SL_ID | 4 byte | 장치 식별자 |
| LEN | 2 byte | Payload 길이 |
| PAYLOAD | N byte | 데이터 영역 |
| CRC16 | 2 byte | 오류 검출 |

### 4.3 메시지 종류

| 코드 | 이름 | 설명 |
|---|---|---|
| `0x01` | Read Request | 데이터 조회 요청 |
| `0x02` | Read Response | 데이터 조회 응답 |
| `0x03` | Control Request | 릴레이 제어 요청 |
| `0x04` | Control Response | 릴레이 제어 응답 |
| `0x05` | Event Report | 이벤트 보고 |
| `0x06` | ACK | 정상 수신 확인 |
| `0x07` | NACK | 오류 응답 |

### 4.4 데이터 항목 정의

#### 지선차단기 -> DCU

| 항목 ID | 항목명 | 타입 | 설명 |
|---|---|---|---|
| `0x0001` | SL_ID | uint32 | 장치 식별자 |
| `0x0002` | Voltage | uint16 | 전압, 단위 0.1 V |
| `0x0003` | Current | uint16 | 전류, 단위 0.01 A |
| `0x0004` | Energy_Acc | uint32 | 누적 전력량, 단위 Wh |
| `0x0005` | Relay_State | uint8 | 릴레이 상태 |
| `0x0006` | Alarm_Code | uint16 bit field | 이상 상태 코드 |

#### DCU -> 지선차단기

| 항목 ID | 항목명 | 타입 | 설명 |
|---|---|---|---|
| `0x1001` | Read_Item_List | list | 조회할 항목 목록 |
| `0x1002` | Relay_Command | uint8 | 릴레이 ON/OFF 명령 |
| `0x1003` | Control_Reason | uint8 | 제어 사유 코드 |
| `0x1004` | Control_Token | uint32 | 제어 명령 검증값 |

### 4.5 Payload 형식

Payload는 `ITEM_ID + TYPE + LEN + VALUE` 구조를 기본으로 한다.

| 필드 | 크기 | 설명 |
|---|---:|---|
| ITEM_ID | 2 byte | 항목 식별자 |
| TYPE | 1 byte | 데이터 타입 |
| ITEM_LEN | 1 byte | 값 길이 |
| VALUE | N byte | 실제 값 |

### 4.6 예시

#### 조회 요청

DCU가 전압, 전류, 릴레이 상태를 요청하는 예시:

```text
[STX][01][01][21][SL_ID][0006][0002 0003 0005][CRC]
```

#### 조회 응답

지선차단기가 전압 223.4V, 전류 10.50A, 릴레이 ON을 응답하는 예시:

```text
[STX][01][02][21][SL_ID][000D]
[0002][uint16][02][08 BA]
[0003][uint16][02][04 1A]
[0005][uint8 ][01][01]
[CRC]
```

#### 릴레이 OFF 제어 요청

```text
[STX][01][03][22][SL_ID][0002][1002 00][CRC]
```

#### 릴레이 OFF 제어 응답

```text
[STX][01][04][22][SL_ID][0004][00][0005 01 00][CRC]
```

위 응답의 의미는 아래와 같다.

- 응답코드 `00`: 제어 성공
- `Relay_State = 00`: 실제 릴레이 상태 OFF

### 4.7 왜 이렇게 설계했는가

#### 4.7.1 간단한 프레임을 쓰는 이유

- MCU 펌웨어 구현이 쉽다.
- DCU 파서가 단순하다.
- 바이트 단위 디버깅이 쉽다.
- 시험 시 원인 분석이 빠르다.

#### 4.7.2 SEQ를 넣는 이유

- 요청과 응답을 정확히 매칭하기 위해 필요하다.
- 재전송 시 중복 응답을 구분하기 위해 필요하다.
- 릴레이 제어처럼 중요한 명령의 이중 수행을 줄일 수 있다.

#### 4.7.3 Relay_State를 별도로 넣는 이유

- "명령을 보냈다"와 "실제로 상태가 바뀌었다"는 다르기 때문이다.
- 제어 결과를 실제 상태값으로 검증해야 운영 신뢰성이 올라간다.

#### 4.7.4 Alarm_Code를 넣는 이유

- 전압, 전류 값만으로 내부 이상 원인을 알기 어렵다.
- 릴레이 구동 실패, 측정칩 이상, 내부 메모리 이상 등을 코드로 분리할 수 있다.
- DCU가 경보 처리와 유지보수 판단을 빠르게 할 수 있다.

#### 4.7.5 CRC16을 넣는 이유

- 통신 잡음이나 프레임 깨짐을 검출하기 위해 필요하다.
- 잘못된 전압값이나 잘못된 제어 명령 해석을 막을 수 있다.

### 4.8 장점

- 구현이 쉽다.
- 프레임이 짧고 단순하다.
- 개발 기간이 짧다.
- 디버깅이 쉽다.
- 필요한 데이터만 정확히 설계할 수 있다.

### 4.9 단점

- 완전한 표준 프로토콜은 아니다.
- 타사 장비와의 직접 호환성은 낮다.
- 장기적으로 표준 연동이 필요하면 다시 확장 설계가 필요할 수 있다.

## 5. 2안: DLMS/COSEM 기반 방식

### 5.1 개념

DLMS/COSEM 기반 방식은 국제 계량 통신에서 널리 사용하는 구조를 따르는 방식이다.

기본 흐름은 아래와 같다.

1. HDLC 링크 연결
2. AARQ/AARE 애플리케이션 세션 수립
3. GET / SET / ACTION 방식으로 객체 조회 및 제어
4. 세션 종료 및 링크 해제

즉, 데이터 하나를 단순히 "항목 ID"로 읽는 것이 아니라, COSEM 객체의 클래스, 논리명, 속성을 통해 접근하는 구조이다.

### 5.2 통신 흐름

대표 흐름 예시는 아래와 같다.

1. `SNRM` 송신
2. `UA` 수신
3. `AARQ` 송신
4. `AARE` 수신
5. `GET Request` 송신
6. `GET Response` 수신
7. 필요 시 `SET` 또는 `ACTION`
8. `Disconnect` 수행

### 5.3 데이터 표현 방식

데이터는 보통 아래 요소로 정의된다.

- `Class ID`
- `OBIS Code`
- `Attribute ID`
- `Data Type`

예를 들어 어떤 값을 읽는다는 것은 아래 의미가 된다.

- 어느 종류의 객체인지
- 어떤 논리명(OBIS)인지
- 그 객체의 몇 번째 속성인지

### 5.4 제안 데이터 매핑 예시

아래는 회의용 개념 예시이며, 실제 OBIS 채택은 표준 검토 후 확정해야 한다.

| 데이터 | 표현 방향 |
|---|---|
| `SL_ID` | Data/Object 식별용 octet-string 또는 logical name 매핑 |
| `Voltage` | Register 또는 Extended Register |
| `Current` | Register 또는 Extended Register |
| `Energy_Acc` | Register 또는 Extended Register |
| `Relay_State` | Disconnect Control 상태 또는 Status 객체 |
| `Alarm_Code` | Status/bit-string 형태의 전용 객체 |

### 5.5 패킷 개념 예시

#### 링크 및 세션 시작

```text
SNRM -> UA
AARQ -> AARE
```

#### 전력량 조회

```text
GET Request(Class ID + OBIS + Attribute)
GET Response(Data Type + Value)
```

#### 릴레이 제어

```text
ACTION Request(Disconnect Control)
ACTION Response(Result)
GET Request(Relay State)
GET Response(Current State)
```

### 5.6 왜 이렇게 설계하는가

#### 5.6.1 표준 구조를 따르는 이유

- 국제적으로 널리 알려진 계량 통신 구조를 활용할 수 있다.
- 상위 시스템 또는 외부 장비와의 확장성 측면에서 유리하다.
- 객체 기반 모델이라 데이터 체계가 명확하다.

#### 5.6.2 세션 절차가 있는 이유

- 단순 바이트 교환이 아니라, 애플리케이션 서비스 사용 전 연결 상태와 조건을 맞추기 위해 필요하다.
- 인증, 컨텍스트 협상, 서비스 호환성 확인에 유리하다.

#### 5.6.3 타입 정보가 포함되는 이유

- 정수, 실수, 문자열, 배열, 구조체 등을 명확히 구분할 수 있다.
- 복잡한 계량 데이터 표현에 적합하다.

#### 5.6.4 객체 기반 접근을 하는 이유

- 데이터가 많아질수록 구조적으로 관리하기 쉽다.
- 전압, 전류, 이력, 이벤트, 제어, 프로파일 데이터까지 체계적으로 확장 가능하다.

### 5.7 장점

- 표준 계열이라 설명력이 좋다.
- 향후 외부 시스템 연동에 유리하다.
- 데이터가 복잡해질수록 체계적으로 확장할 수 있다.
- 보안, 세션, 객체 모델 등 확장 여지가 크다.

### 5.8 단점

- 구현 복잡도가 높다.
- 펌웨어 메모리와 코드량 부담이 커진다.
- DCU 쪽 해석 로직이 복잡하다.
- 현장 디버깅이 어렵다.
- 지금 요구 범위에는 다소 무거울 수 있다.

## 6. 두 안의 비교

| 항목 | 1안: 간단한 전용 방식 | 2안: DLMS/COSEM 방식 |
|---|---|---|
| 구조 | 단순 요청/응답 | 링크 + 세션 + 객체 접근 |
| 구현 난이도 | 낮음 | 높음 |
| 디버깅 | 쉬움 | 어려움 |
| 데이터 확장 | 보통 | 우수 |
| 표준성 | 낮음 | 높음 |
| 타 시스템 연계 | 별도 개발 필요 | 상대적으로 유리 |
| MCU 부담 | 낮음 | 높음 |
| 현재 요구 적합성 | 높음 | 중간 |
| 장기 확장성 | 중간 | 높음 |

## 7. 회의에서 설명할 핵심 포인트

### 7.1 1안을 설명할 때

아래와 같이 설명할 수 있다.

- 지금 필요한 데이터 종류가 많지 않기 때문에, 필요한 항목만 읽고 제어하는 경량 방식이 적합하다.
- 릴레이 제어와 상태 확인이 핵심이므로, 복잡한 객체 체계보다 단순하고 확실한 응답 구조가 중요하다.
- 개발 기간, 디버깅, 시험 대응, 유지보수 측면에서 유리하다.
- 나중에 항목이 늘어나더라도 항목 ID를 추가하는 방식으로 확장 가능하다.

### 7.2 2안을 설명할 때

아래와 같이 설명할 수 있다.

- 기존 계량기 통신처럼 표준 계열 구조를 따르려면 DLMS/COSEM 방식이 맞다.
- 장기적으로 외부 시스템 연동, 표준화, 복잡한 데이터 모델까지 고려하면 유리하다.
- 다만 현재 요구 수준에서는 구현이 무겁고, 개발 및 시험 부담이 크다.

### 7.3 권고안 설명

아래와 같이 정리할 수 있다.

- 초기 양산 또는 시제품 단계에서는 1안이 더 현실적이다.
- 이후 외부 표준 연계 필요성이 커지면 2안 또는 1안의 DLMS 확장형을 검토할 수 있다.
- 즉, 당장은 간단한 전용 프로토콜로 시작하고, 구조는 DLMS처럼 확장 가능하게 설계하는 것이 균형이 좋다.

## 8. 최종 권고

현 시점에서는 `1안: 간단한 전용 프로토콜 방식`을 우선 채택하는 것을 권고한다.

그 이유는 다음과 같다.

- 현재 필요한 데이터 수가 제한적이다.
- 운영 핵심은 표준성보다 안정적인 수집과 제어 확인이다.
- 릴레이 상태와 알람 코드 같은 운영용 데이터 정의가 중요하다.
- 개발 및 검증 리스크를 낮출 수 있다.

다만 문서 구조와 데이터 체계는 DLMS처럼 항목 기반, 확장형으로 설계하여 향후 확장성을 확보하는 것이 바람직하다.
