# 지선차단기 - DCU 통신 프로토콜 초안

작성일: 2026-04-09

## 1. 목적

본 문서는 지선차단기와 DCU 간의 1차 통신 프로토콜 초안을 정의한다.

이번 초안의 목표는 다음과 같다.

- 지선차단기의 계측값과 상태 정보를 DCU가 안정적으로 수집할 수 있어야 한다.
- DCU가 지선차단기에 필요한 제어 명령을 안전하게 전달할 수 있어야 한다.
- 전체 구조는 DLMS/COSEM의 방향성을 참고하되, 초기 개발과 시험에 적합하도록 경량화한다.

본 문서는 완전한 DLMS/COSEM 구현이 아니라, 아래 개념을 참고한 전용 경량 프로토콜을 대상으로 한다.

- 클라이언트/서버 구조
- 객체 또는 항목 기반 데이터 식별
- 요청/응답 구조
- 향후 확장 가능한 프레임 구조

## 2. 적용 범위

본 초안은 초기 버전의 지선차단기와 DCU 간 통신에 적용한다.

- 지선차단기 역할: 계측값 및 상태 제공, 제어 명령 수행
- DCU 역할: 데이터 수집, 상태 확인, 릴레이 제어 명령 송신

이번 버전에서 장치 시간 동기화는 프로토콜 항목에서 제외한다.

- 이유: 시간 기준은 DCU 수신 시각을 기준으로 관리한다.
- 따라서 지선차단기에서 별도 시간 값을 송신하지 않는다.

## 3. 설계 방향

### 3.1 기본 방향

- 통신 주체는 `DCU = Client(Master)`, `지선차단기 = Server(Slave)`로 정의한다.
- DCU는 주기적으로 지선차단기를 조회한다.
- 지선차단기는 조회 요청에 응답하며, 필요 시 이벤트를 자발 보고할 수 있도록 구조를 열어둔다.

### 3.2 DLMS 참고 범위

DLMS/COSEM의 모든 계층과 객체 체계를 그대로 구현하지는 않는다. 대신 아래 항목만 반영한다.

- 데이터 항목별 식별자 부여
- 요청/응답 기반 통신
- 향후 확장을 위한 버전 필드 유지
- 상태/이상/제어 결과를 분리하여 표현

이 방식은 초기 개발 난이도와 구현 자원 부담을 줄이면서도, 나중에 항목 확장과 상위 시스템 연계를 쉽게 한다.

## 4. 통신 방식 개요

### 4.1 기본 동작

기본 통신 흐름은 아래와 같다.

1. DCU가 지선차단기에 조회 요청을 송신한다.
2. 지선차단기가 계측값 및 상태를 응답한다.
3. DCU가 필요 시 릴레이 제어 명령을 송신한다.
4. 지선차단기가 제어 수행 결과를 응답한다.

### 4.2 메시지 종류

- `Read Request`: 데이터 조회 요청
- `Read Response`: 데이터 조회 응답
- `Control Request`: 제어 명령 요청
- `Control Response`: 제어 명령 응답
- `Event Report`: 이벤트 자발 보고
- `ACK`: 정상 수신 확인
- `NACK`: 비정상 수신 또는 처리 실패 통보

## 5. 프레임 구조

### 5.1 기본 프레임

초기 버전의 기본 프레임 구조는 다음과 같이 정의한다.

| 필드 | 크기 | 설명 |
|---|---:|---|
| STX | 1 byte | 프레임 시작 표시 |
| VER | 1 byte | 프로토콜 버전 |
| MSG_TYPE | 1 byte | 메시지 종류 |
| SEQ | 1 byte | 시퀀스 번호 |
| SL_ID | 4 byte | 지선차단기 식별자 |
| LEN | 2 byte | Payload 길이 |
| PAYLOAD | N byte | 실제 데이터 |
| CRC16 | 2 byte | 오류 검출용 CRC |

### 5.2 필드별 필요 이유

#### STX

- 프레임의 시작 위치를 찾기 위해 필요하다.
- 잡음 또는 수신 깨짐이 생겼을 때 수신기 재동기를 쉽게 해준다.

#### VER

- 프로토콜 버전 변경 시 상호 호환 여부를 판단하기 위해 필요하다.
- 초기 버전 이후 필드 추가나 메시지 확장 시 필수적이다.

#### MSG_TYPE

- 같은 프레임 구조 안에서 조회, 응답, 제어, 이벤트를 구분하기 위해 필요하다.
- 펌웨어와 DCU 로직을 단순하게 분리할 수 있다.

#### SEQ

- 요청과 응답이 서로 짝이 맞는지 확인하기 위해 필요하다.
- 재전송이 발생했을 때 중복 응답 또는 오래된 응답을 구분할 수 있다.
- 제어 명령의 경우 "현재 응답이 방금 보낸 명령에 대한 결과인지"를 확인하는 기준이 된다.

#### SL_ID

- 여러 장치가 연결되는 환경에서 어떤 지선차단기의 데이터인지 구분하기 위해 필요하다.
- 장치 주소와 상위 시스템 식별 정보를 일치시키는 기준이 된다.

#### LEN

- Payload 길이를 정확히 판단하기 위해 필요하다.
- 가변 길이 데이터 구조에서 다음 프레임 경계를 안정적으로 찾을 수 있다.

#### PAYLOAD

- 실제 계측값, 상태값, 제어값, 에러코드 등 운용 데이터를 담는 영역이다.

#### CRC16

- 통신 중 비트 오류, 잡음, 프레임 손상을 검출하기 위해 필요하다.
- 계측값이나 제어 명령이 잘못 해석되는 위험을 줄인다.

## 6. 메시지 코드 정의

| 코드 | 이름 | 설명 |
|---|---|---|
| `0x01` | Read Request | DCU가 데이터 조회 요청 |
| `0x02` | Read Response | 지선차단기가 데이터 응답 |
| `0x03` | Control Request | DCU가 제어 명령 요청 |
| `0x04` | Control Response | 지선차단기가 제어 결과 응답 |
| `0x05` | Event Report | 지선차단기가 이벤트 자발 보고 |
| `0x06` | ACK | 정상 수신 확인 |
| `0x07` | NACK | 오류 또는 거부 응답 |

## 7. 데이터 모델

### 7.1 데이터 항목 정의 원칙

초기 버전은 복잡한 계층형 객체 대신, `항목 ID + 타입 + 값` 구조를 사용한다.

각 데이터 항목은 고유 ID를 가지며, 향후 항목 추가 시에도 기존 구조를 유지한다.

### 7.2 지선차단기에서 DCU로 보내는 데이터

| 항목 ID | 항목명 | 타입 예시 | 단위 | 필수 여부 | 설명 |
|---|---|---|---|---|---|
| `0x0001` | SL_ID | uint32 | - | 필수 | 지선차단기 식별자 |
| `0x0002` | Voltage | uint16 | 0.1 V | 필수 | 실효 전압 |
| `0x0003` | Current | uint16 | 0.01 A | 필수 | 실효 전류 |
| `0x0004` | Energy_Acc | uint32 | Wh | 필수 | 누적 전력량 |
| `0x0005` | Relay_State | uint8 | - | 필수 | 릴레이 현재 상태 |
| `0x0006` | Alarm_Code | uint16 bit field | - | 권장 | 이상 상태 코드 |
| `0x0007` | Instant_Power | int32 | W | 선택 | 순간 유효전력 |
| `0x0008` | Device_Status | uint16 bit field | - | 권장 | 통신 외 내부 상태 |
| `0x0009` | FW_Version | string or uint32 | - | 선택 | 펌웨어 버전 |

### 7.3 DCU에서 지선차단기로 보내는 데이터

| 항목 ID | 항목명 | 타입 예시 | 필수 여부 | 설명 |
|---|---|---|---|---|
| `0x1001` | Read_Item_List | list | 필수 | 조회할 항목 목록 |
| `0x1002` | Relay_Command | uint8 | 필수 | 릴레이 ON/OFF 명령 |
| `0x1003` | Control_Reason | uint8 | 권장 | 제어 사유 코드 |
| `0x1004` | Control_Token | uint32 or uint64 | 권장 | 제어 명령 유효성 확인값 |
| `0x1005` | Polling_Profile | uint8 | 선택 | 조회 주기/운영 모드 지정 |
| `0x1006` | Request_Mask | uint16 bit field | 선택 | 기본 조회 세트 선택 |

## 8. 각 데이터가 필요한 이유

### 8.1 SL_ID

- 어떤 장치의 데이터인지 구분하는 기본 키이기 때문에 필수이다.
- DCU, 서버, 운영 화면이 모두 같은 장치를 동일하게 인식할 수 있어야 한다.
- 설치, 교체, 유지보수 이력 관리의 기준이 된다.

### 8.2 Voltage

- 전원 상태의 정상 여부를 판단하기 위해 필요하다.
- 저전압, 과전압, 무전압 상태를 판별하는 기본 값이다.
- 장치 이상인지, 부하 이상인지, 전원 계통 이상인지 구분하는 출발점이 된다.

### 8.3 Current

- 실제 부하가 연결되어 동작 중인지 판단하기 위해 필요하다.
- 과전류, 무부하, 예상과 다른 소비 상태를 확인할 수 있다.
- 릴레이 제어 후 실제 차단 또는 투입 결과를 간접 검증하는 데도 도움이 된다.

### 8.4 Energy_Acc

- 사용량 정산, 이력 분석, 누적 소비량 추적을 위해 필요하다.
- 순간값만으로는 알 수 없는 장기 사용 패턴을 볼 수 있다.
- 통신 누락이 일부 발생해도 누적값 기준으로 보정이나 추적이 가능하다.

### 8.5 Relay_State

- 지선차단기가 현재 투입 상태인지 차단 상태인지 즉시 확인하기 위해 필요하다.
- 제어 명령을 보냈을 때 실제 결과가 반영되었는지 확인하는 핵심 값이다.
- 운영자는 "명령을 보냈다"보다 "실제로 릴레이가 어떤 상태인가"를 더 중요하게 본다.

### 8.6 Alarm_Code

- 단순 계측값만으로 표현되지 않는 이상 상태를 명확히 전달하기 위해 필요하다.
- 예를 들어 과전류, 센서 이상, 릴레이 접점 이상, 메모리 오류, 통신 이상 등을 코드로 구분할 수 있다.
- DCU나 서버가 즉시 경보 정책을 적용하거나 유지보수 우선순위를 정하는 데 유용하다.

알람 코드가 없으면, 운영 측은 전압/전류 값만 보고 상황을 추정해야 한다. 이는 해석 오류와 대응 지연으로 이어질 수 있다.

### 8.7 Instant_Power

- 현재 부하 소비 수준을 빠르게 판단하는 데 유용하다.
- 초기 버전 필수는 아니지만, 운영 화면과 분석 기능에서 활용도가 높다.

### 8.8 Device_Status

- 통신은 살아 있어도 내부 이상이 있을 수 있으므로 별도 상태 비트가 있으면 좋다.
- 예를 들어 계량칩 오류, EEPROM 오류, 릴레이 구동 실패, 보정값 이상 등을 표현할 수 있다.

### 8.9 FW_Version

- 장치별 펌웨어 버전을 파악해 장애 재현, 원인 분석, 배포 관리에 활용할 수 있다.
- 필수는 아니지만 현장 운영 단계에서 매우 도움이 된다.

### 8.10 Read_Item_List

- DCU가 필요한 항목만 선택 조회할 수 있게 해준다.
- 통신량을 줄이고, 상황에 맞는 조회 구성이 가능해진다.

### 8.11 Relay_Command

- 원격 차단/투입 기능의 핵심 제어 항목이다.
- 본 시스템의 제어 목적을 수행하기 위해 필수이다.

### 8.12 Control_Reason

- 왜 제어했는지 기록과 해석을 돕기 위해 필요하다.
- 예를 들어 수동 제어, 스케줄 제어, 보호 제어, 원격 운영 명령 등을 구분할 수 있다.
- 나중에 로그 분석 시 "누가 왜 차단했는가"를 이해하는 데 도움이 된다.

### 8.13 Control_Token

- 제어 명령의 유효성을 확인하는 간단한 보호 장치로 사용할 수 있다.
- 초기 버전에서 복잡한 보안을 다 넣지 않더라도, 최소한의 오동작 방지 수단이 된다.

### 8.14 Polling_Profile

- DCU가 평상시/장애시/시험시 등 운영 모드에 따라 조회 정책을 구분할 수 있다.
- 필수는 아니지만 향후 확장성이 좋다.

### 8.15 Request_Mask

- 자주 쓰는 조회 묶음을 비트 단위로 간단히 지정할 수 있다.
- 메시지를 짧게 유지하는 데 도움이 된다.

## 9. Payload 기본 형식

Payload는 아래의 TLV 유사 구조를 기본으로 한다.

| 필드 | 크기 | 설명 |
|---|---:|---|
| ITEM_ID | 2 byte | 데이터 항목 식별자 |
| TYPE | 1 byte | 데이터 타입 |
| ITEM_LEN | 1 byte | 값 길이 |
| VALUE | N byte | 데이터 값 |

예시:

| 항목 | 값 예시 |
|---|---|
| Voltage | `ITEM_ID=0x0002`, `TYPE=uint16`, `VALUE=2234` |
| Current | `ITEM_ID=0x0003`, `TYPE=uint16`, `VALUE=1050` |
| Relay_State | `ITEM_ID=0x0005`, `TYPE=uint8`, `VALUE=1` |

스케일 정의 예:

- Voltage `2234` = `223.4 V`
- Current `1050` = `10.50 A`

## 10. 릴레이 상태 정의

| 값 | 의미 |
|---|---|
| `0` | OFF / 차단 |
| `1` | ON / 투입 |
| `2` | 전환 중 |
| `3` | 이상 상태 / 판별 불가 |

### 10.1 릴레이 상태가 꼭 필요한 이유

- 릴레이는 본 장치의 핵심 제어 대상이므로 현재 상태가 반드시 보여야 한다.
- DCU가 ON 명령을 보냈더라도, 실제 접점이 붙지 않았을 수 있다.
- 반대로 OFF 명령 후에도 접점 용착 등으로 차단 실패가 발생할 수 있다.
- 따라서 제어 명령 자체와 별도로 "실제 상태값"을 반드시 읽어야 한다.

## 11. 알람 코드 정의 초안

초기 버전에서는 `bit field` 방식을 권장한다.

| 비트 | 의미 |
|---:|---|
| bit0 | 과전류 |
| bit1 | 과전압 |
| bit2 | 저전압 |
| bit3 | 릴레이 구동 실패 |
| bit4 | 계량칩 이상 |
| bit5 | 내부 메모리 이상 |
| bit6 | 센서 또는 측정값 이상 |
| bit7 | 예약 |
| bit8~15 | 향후 확장 |

### 11.1 알람 코드를 별도로 두는 이유

- 전압/전류 수치만으로는 장치 내부 이상을 충분히 표현할 수 없다.
- 운영 측은 "현재 값"과 함께 "이상이 무엇인지"를 빠르게 알아야 한다.
- 비트 필드 방식은 메시지가 짧고, 여러 이상을 동시에 표현할 수 있다.

## 12. 시퀀스 번호 정의

`SEQ`는 1 byte 순환 값으로 정의한다.

- 범위: `0x00` ~ `0xFF`
- 송신 시 1씩 증가
- `0xFF` 이후 `0x00`으로 순환

### 12.1 시퀀스 번호가 필요한 이유

- 동일한 요청의 재전송 여부를 판별할 수 있다.
- 응답이 늦게 도착했을 때 이전 요청의 응답인지 확인할 수 있다.
- 특히 릴레이 제어처럼 중요한 명령에서 응답 매칭이 정확해진다.

예를 들어:

- DCU가 `SEQ=0x21`로 릴레이 OFF 명령 송신
- 응답 지연 중 timeout 발생
- DCU가 재전송
- 지선차단기는 같은 `SEQ`를 보고 중복 명령 여부를 판단 가능

이 구조가 없으면 같은 명령이 두 번 수행될 위험이 커진다.

## 13. 응답 코드 정의

Control Response 또는 NACK에서 아래 응답 코드를 사용할 수 있다.

| 코드 | 의미 |
|---|---|
| `0x00` | 성공 |
| `0x01` | 알 수 없는 메시지 |
| `0x02` | 지원하지 않는 항목 |
| `0x03` | 길이 오류 |
| `0x04` | CRC 오류 |
| `0x05` | 잘못된 값 |
| `0x06` | 제어 거부 |
| `0x07` | 내부 처리 실패 |
| `0x08` | 인증 또는 토큰 오류 |

### 13.1 응답 코드가 필요한 이유

- 실패를 단순히 "안 됨"으로만 처리하면 원인 파악이 어렵다.
- 설치, 시험, 유지보수 시 문제 지점을 빠르게 찾을 수 있다.
- DCU가 재시도할지, 경보를 올릴지, 운영자 확인이 필요한지 판단할 수 있다.

## 14. 권장 통신 시나리오

### 14.1 주기 조회

- DCU는 주기적으로 `Read Request`를 전송한다.
- 기본 조회 항목:
  - `SL_ID`
  - `Voltage`
  - `Current`
  - `Energy_Acc`
  - `Relay_State`
  - `Alarm_Code`

### 14.2 제어 수행

1. DCU가 `Control Request` 전송
2. 지선차단기가 명령 형식 및 토큰 검증
3. 릴레이 동작 수행
4. `Control Response`에서 결과 반환
5. 필요 시 직후 `Read Request` 또는 응답 내 상태 포함으로 최종 상태 확인

### 14.3 이벤트 보고

향후 확장을 고려하여 아래 이벤트는 `Event Report` 대상으로 설계 가능하다.

- 릴레이 상태 변경
- 알람 발생
- 알람 해제
- 재기동

초기 버전에서는 우선 조회 기반 운용을 기본으로 하고, 이벤트 보고는 선택 기능으로 둔다.

## 15. 초기 버전 필수 항목 정리

초기 버전에서 반드시 포함하는 항목은 아래와 같다.

- 프레임: `STX`, `VER`, `MSG_TYPE`, `SEQ`, `SL_ID`, `LEN`, `PAYLOAD`, `CRC16`
- 지선차단기 송신 데이터: `SL_ID`, `Voltage`, `Current`, `Energy_Acc`, `Relay_State`
- 권장 포함 데이터: `Alarm_Code`
- DCU 송신 데이터: `Read_Item_List`, `Relay_Command`
- 권장 제어 보조 데이터: `Control_Reason`, `Control_Token`
- 공통 처리: 응답 코드 정의, timeout 및 재전송 정책

## 16. 초기 버전에서 제외하는 항목

이번 버전에서 제외하는 항목은 아래와 같다.

- 지선차단기 장치 시간 값
- 시간 동기화 명령
- 완전한 DLMS OBIS 체계
- 복잡한 세션 관리
- 인증서 기반 보안
- 대용량 블록 전송

## 17. 추후 확장 항목

향후 필요 시 아래 항목을 추가 검토한다.

- 순간 전력, 역률, 주파수
- 이벤트 로그
- 제어 이력 번호
- 장치 시간 및 시간 품질
- 보안 인증 강화
- 그룹 제어 또는 브로드캐스트

## 18. 결론

지선차단기 - DCU 통신의 초기 버전은 DLMS의 구조적 장점을 참고하되, 현장 적용과 개발 효율을 위해 경량 전용 프로토콜로 설계하는 것이 적절하다.

특히 아래 항목은 단순해 보여도 실제 운영 안정성에 직접 영향을 주므로 반드시 포함하는 것이 바람직하다.

- `Relay_State`: 제어 결과의 실제 상태 확인용
- `Alarm_Code`: 수치값만으로 설명되지 않는 이상 상태 전달용
- `SEQ`: 요청/응답 매칭 및 재전송 중복 방지용
- `CRC16`: 통신 오류 검출용

이 초안을 기준으로 다음 단계에서는 아래 문서를 연계 작성할 수 있다.

- 패킷 예시 정의
- 상태 전이도
- timeout / retry 정책
- 시험 항목 및 검증 기준
