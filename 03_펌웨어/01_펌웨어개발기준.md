## 3. 기준 인터페이스

### 3.1 최종 제품 기준

- `ATM90E26-YU`: `SPI1`
- `DCU 통신`: `USART2 + RS-485`
- `서비스/로그`: `USART1`
- `릴레이`: GPIO drive + feedback

### 3.2 권장 핀

| 구간 | 방식 | 권장 핀 | 비고 |
|---|---|---|---|
| MCU ↔ ATM90E26 | SPI1 | `PA5`, `PA6`, `PA7`, `PB6`, `PB7` | `SCK/MISO/MOSI/CS/RESET` |
| MCU ↔ DCU | USART2 + RS-485 | `PA2`, `PA3`, `PA1` | `TX/RX/DE-RE` |
| MCU ↔ Relay | GPIO | `PB0`, `PB1` | `Drive/Feedback` |
| 서비스 UART | USART1 | `PA9`, `PA10` | 디버그 / 로그 |
| 디버깅 포트 | SWD | `NRST`, `PA13`, `PA14` | `RESET/SWDIO/SWCLK` |
| ATM90 확장핀 | GPIO/EXTI | `PB10~PB14` | `IRQ/WarnOut/ZX/CF1/CF2` |

## 4. LCU 보드 수령 후 즉시 테스트 절차

### 4.1 목표

- 보드 수령 당일 `LCU 보드`로 데모가 가능한지 판정한다.
- 가능하면 같은 날 `ATM90E26 기본 register read`까지 진입한다.

### 4.2 준비물

- `LCU 보드`
- `전원 공급 장치`
- `ST-LINK` 또는 보드 다운로드/디버깅 장치
- `RS-485 to USB` 컨버터
- PC
- 점퍼선 / 프로브 / 멀티미터
- 가능한 경우 회로도 / 소스코드 / 빌드 환경 정보

### 4.3 보드 수령 직후 확인 순서

1. 보드 외관 확인
2. 전원 입력 위치 확인
3. MCU 다운로드 포트 확인
4. 디버깅 포트 확인
5. ATM90E26와 MCU 연결 방식 확인
6. 6채널 중 1채널 우선 테스트 가능 여부 확인
7. PC와 연결할 통신 경로 확인

## 6. LCU 보드 당일 테스트 순서

### 6.1 1단계: 개발 환경 확인

1. 보드 전원 인가
2. ST-LINK 또는 디버거 연결
3. 기존 펌웨어 읽기 또는 프로젝트 열기
4. 빌드/다운로드 가능한지 확인

### 6.2 2단계: ATM90E26 인터페이스 확인

1. `SPI` 또는 `UART` 연결 확인
2. `CS`, `RESET`, `SCK/MISO/MOSI` 또는 `TX/RX` 경로 확인
3. 테스트용 채널 1개 선택

### 6.3 3단계: 기본 통신 확인

1. ATM90 reset 제어
2. 기본 register read
3. register write/read back 확인

### 6.4 4단계: 계측값 read

1. 전압 read
2. 전류 read
3. 유효전력 read
4. 주파수 read
5. 가능하면 무효전력 / 피상전력 / 역률 read

### 6.5 5단계: 데모 경로 확정

- 계측값 read가 성공하면 `LCU 보드 경로 유지`
- 기본 통신이 불안정하거나 개발 환경이 막히면 `Eval 보드`로 전환

- `LCU 보드 데모`와 별개로 `STM32L053R8T6 최종 보드 준비`를 병렬로 진행한다.
- 목표는 최종 보드가 도착했을 때 `transport 연결`만 추가하고 바로 이식하는 것이다.

### 10.1 지금 미리 준비할 것

1. `STM32L053R8T6` 최종 핀맵 고정
2. `CubeMX` 프로젝트 생성 준비
3. `SPI1 + USART2 + USART1 + Relay GPIO + SWD` 기준 초기 구조 정리
4. `ATM90E26 transport`를 보드 종속 코드와 분리 유지
5. `RS-485 polling`, `보호 로직`, `복구 로직`은 보드 비종속 코드로 계속 개발

### 10.2 보드 도착 전 완료 목표
